# Spring Security 정리

## 1. Spring Security

Spring 기반 애플리케이션의 ` 인증 + 인가 ` 을 담당하는 하위 프레임워크.

핵심은 **Filter 흐름으로 처리**한다는 것.

- 요청이 들어오면 `DispatcherServlet` 가기 전에 필터에서 먼저 가로챔
- Interceptor는 Dispatcher ~ Controller 사이 → **적용 시기가 다름**
- 보안 로직을 직접 다 짤 필요 없이 옵션으로 제공됨

### 필터 동작 원리

```
클라이언트 요청 → 서블릿 필터 → 서블릿(컨트롤러)
```

1. 톰캣 필터 체인에 `DelegatingFilterProxy` 등록 (톰캣은 스프링 빈을 실행 못 하니까) →<br> **모든 요청 가로챔**
2. 가로챈 요청을 `FilterChainProxy`으로 넘김
3. 여기서 인증/인가 검증 후 → 거부 / 리디렉션 / 컨트롤러 전달 결정

**doFilter**<br> 
doFilter를 호출하면 다음 필터 안쪽으로 들어갔다가, 다 끝나면 다시 바깥으로 빠져나온다.

## 2. JWT / 세션 / 쿠키

### 쿠키 (Cookie)
데이터를 저장하고 전송하는 수단
(` 브라우저가 자동으로 들고 다님 `)

### 세션 (Session)
상태를 서버가 기억하는 방식

### JWT (Json Web Token)

사용자 정보를 토큰 자체에 담는 방식(` 상태를 서버가 안 들고 있음 `  )

**JWT 구조: `Header . Payload . Signature`**

| 부분 | 내용 |
|------|------|
| Header | 토큰 타입, 서명 알고리즘 (예: HS256) |
| Payload | 실제 정보(클레임). username, role, 발급일, 만료일 등 |
| Signature | Header + Payload를 secret key로 암호화 → **변조 방지** |

**변조 검증 원리**: 누가 Payload를 조작해도, Signature는 조작 전 내용 기반으로 이미 암호화돼 있어서 → 서버가 다시 계산하면 값이 안 맞음 → 조작 감지 가능.

## 3. Spring Security 주요 구성 요소


### 인증 흐름 (로그인)

```
LoginFilter → AuthenticationManager → UserDetailsService → SecurityContextHolder
```

| 구성 요소 | 역할 | 일 |
|-----------|------|-------|
| SecurityFilterChain | 설정 | 뭘 어떻게 보호할지 규칙 정의 |
| SecurityContextHolder | 저장 | 현재 로그인 사용자 보관 (ThreadLocal) |
| Authentication | 데이터 | 인증 주체 (누구인지 + 권한) |
| UserDetailsService | 조회 | DB에서 사용자 가져옴 |
| PasswordEncoder | 검증 | 비밀번호 암호화·비교 (BCrypt) |

### 로그인 성공 시
`successfulAuthentication()` 에서 JWT 발급 → 응답 헤더에 담음:
```
Authorization: Bearer {토큰}
```
실패 시 `unsuccessfulAuthentication()` 에서 `401` 반환.

### 요청 인가 시 (JWTFilter)
1. `Authorization` 헤더 확인 → `Bearer ` 없으면 통과(다음 필터)
2. `Bearer` 제거하고 순수 토큰 추출
3. 만료 검증
4. 토큰에서 username, role 꺼내서 → SecurityContextHolder에 인증 정보 등록
5. STATELESS라 서버에 세션을 저장하지 않음 → 요청 끝나면 SecurityContext 소멸

## 4. 비밀번호 암호화

비밀번호는 **평문 저장 절대 X** → 해싱해서 저장.

### BCrypt 특징
- **단방향 해시** → 복호화 불가
- 같은 비밀번호라도 **매번 다른 해시값** 생성 (salt 자동 적용)
- 검증은 `matches(입력값, 저장된해시)` 로 비교 (직접 복호화 안 함)
