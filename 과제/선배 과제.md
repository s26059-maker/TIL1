# 트랜잭션 · 프록시 · ACID 정리

## @Transactional

**트랜잭션(Transaction)** 은 쪼개면 안 되는 작업 한 덩어리라는 뜻이다.

`@Transactional`을 붙이면 스프링이 내 서비스 앞에 **프록시** 를 세워둔다.
**프록시(Proxy)** 는 진짜인 척하는 가짜 객체다.
그래서 컨트롤러가 부르는 건 사실 내 서비스가 아니라 가짜다.

프록시가 하는 일:

```
1. 요청을 먼저 받는다
2. 커넥션 잡고 autocommit 끄기
3. 진짜 내 코드 실행
4. 잘 끝나면 commit / RuntimeException 터지면 rollback
5. 커넥션 반납
```

내가 commit/rollback 코드를 안 짜도 되는 이유 = **프록시가 대신 해주니까.**

### 주의
 자기 자신 호출은 트랜잭션이 안 걸린다

```java
public void outer() {
    inner();   // 트랜잭션 안 걸림!!
}

@Transactional
public void inner() { ... }
```

왜? 트랜잭션은 **프록시를 거쳐야** 걸리는데,
같은 클래스 안에서 부르는 건 프록시 안 거치고 `this`로 **직접** 부르는 거라서.

같은 이유로 `private`, `final` 메서드도 안 됨.
(프록시는 내 클래스를 **상속**해서 만들어지는데, private/final은 오버라이드가 안 되니까)

### 주의
아무 예외나 롤백되는 게 아니다

- `RuntimeException`, `Error` → 롤백 O
- 체크 예외(`Exception`) → **롤백 X, 그냥 커밋됨**
- 체크 예외도 롤백하고 싶으면 → `@Transactional(rollbackFor = Exception.class)`

## 지연로딩

**지연로딩(Lazy Loading)** 은
진짜 필요할 때까지 DB 조회를 미루는 것이다.

### 주의
 여기서 프록시는 위에 나온 프록시랑 다른 애임

- 스프링 프록시 = 트랜잭션 걸어주는 **가짜 객체** (스프링이 만듦)
- JPA 프록시 = 지연로딩용 **가짜 엔티티** (하이버네이트가 만듦)



### 책상 / 창고 비유

영속성 컨텍스트 = 책상, DB = 창고라고 하면:

```java
@ManyToOne(fetch = FetchType.LAZY)
private Team team;
```

Member를 책상으로 가져올 때, Team은 창고에서 안 꺼내온다.
대신 **"Team"이라고 이름표만 붙은 빈 상자**를 놔둔다.

```java
Member member = memberRepository.findById(1L);  
Team team = member.getTeam();                   
team.getName();                                 
```

상자를 여는 순간 창고(DB)에서 진짜를 꺼내와 채우는 것 = **프록시 초기화**.

### 중요

`@Transactional`과 지연로딩의 관계는<br>
지연로딩은 "미루기"라는 목표고, 프록시는 그걸 구현하는 수단이다.

> **트랜잭션 범위 = 영속성 컨텍스트가 살아있는 범위 = 지연로딩이 되는 범위**

## ACID

**ACID**는 트랜잭션이 지켜야 하는 4가지 약속의 앞글자다.<br>
만약 A가 B한테 10,000원 이체한다고 가정하면

**A — Atomicity (원자성)**
atom = 더 쪼갤 수 없는 것.
출금은 됐는데 입금이 실패하면? → 출금까지 취소(롤백).
**전부 되거나, 전부 안 되거나.** 절반만 된 상태는 존재하면 안 됨.

**C — Consistency (일관성)**
이체 전이든 후든 **두 사람 돈의 합은 같아야** 한다.
잔액 ≥ 0, FK 같은 DB 규칙이 트랜잭션 후에도 항상 지켜져야 한다는 뜻.

**I — Isolation (격리성)**
Isolation = '격리, 분리'라는 뜻.
내가 이체하는 **중간 상태를 남이 보면 안 된다.**
동시에 도는 트랜잭션끼리 서로 간섭 금지.

**D — Durability (지속성)**
Durability = '내구성, 오래감'이라는 뜻.
commit 직후 서버가 꺼져도 **이체된 만원은 살아있어야** 한다.
(DB가 로그를 먼저 남겨놔서 장애 나도 복구함 — redo log)

### 격리성

동시에 도는 트랜잭션끼리 서로 간섭하면 안 된다는 약속

완벽하게 격리하면 트랜잭션을 한 줄로 세워야 해서 너무 느림.
그래서 **격리 수준** 4단계로 타협


- READ UNCOMMITTED 
    - 커밋 안 된 것도 보임 (거의 안 씀) 
- READ COMMITTED
    - 커밋된 것만 보임 
- **REPEATABLE READ** 
    - 트랜잭션 안에선 같은 걸 읽으면 항상 같은 값 → **MySQL 기본값**
- SERIALIZABLE 
    - 완벽 격리, 대신 제일 느림

`@Transactional` = **DB가 가진 ACID를 코드에서 편하게 쓰게 해주는 도구.**
커밋/롤백 자동화가 원자성, isolation 옵션이 격리성으로 연결됨.
